Waterfall model

I don't want to go through waterfall as all of us might know, but just to point out some of the problems

Waterfall problems

  • No working software is produced until late.
  • High amounts of risk and uncertainty.
  • It is difficult to measure progress within stages.
  • Cannot accommodate changing requirements.
  • Integration is done as a "big-bang. at the very end, which doesn't allow identifying any technological or business bottleneck or challenges early.

Manifesto for Agile Software Development

We are uncovering better ways of developing software by doing it and helping others do it. Through this work we have come to value:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

That is, while there is value in the items on the right, we value the items on the left more.

  • Working together effectively. A fool with a tool is still a fool.
  • Show me software not documents
  • Working close with the customer or Product Owner in daily basis ( and this I think is violated somehow in Scrum, after I will dig into it
  • Change is a reality of software development. You need a project plan, in fact I would be worried about any project that didn’t have one. But has to be flexible, has to be soft and allow changes

The Nature of Software Development

  • Organising By Feature
  • Planning Feature by Feature
  • Building the product, feature by feature
  • Slice features thinly
  • Build quality in every day
  • Organise teams around the features we need
  • Plan in term of features. We produce highest value first.
  • Build based on features, including the architecture and infrastructure we need as we go
  • Slice each desired capability down to the smallest possible value-bearing size
  • Build value continuously, sustainably, indefinitely.

Slicing heuristic

“A user story ready to be worked on must have only one acceptance test." - Neil Killick
  • Slice features till the smallest possible value-bearing till one acceptance test for feature.
  • Then all features will have a similar size.
  • If u need a speed of a team in each sprint just count features
